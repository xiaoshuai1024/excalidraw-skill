# 10 种常见图的专业画法标准

记录每种图的标准元素、布局惯例和常见错误。编写 gen-diagrams.py 时对照本文档。

## 1. 业务架构图
- **分层布局**：自顶向下或自左向右，每层一个分组区域（虚线框）
- **元素**：矩形框（业务能力/服务），层间箭头（依赖关系）
- **常见错误**：层内元素太多（>6个框挤在一起）

## 2. 部署架构图
- **节点**：矩形框，标注容器/Pod/实例名
- **连接**：箭头表示流量/依赖关系
- **分组**：虚线区域（K8s Cluster / DMZ / VPC）
- **常见错误**：缺少负载均衡/入口网关

## 3. 流程图（Flowchart）
- **开始/结束**：**椭圆（ellipse）**
- **处理步骤**：**矩形（rectangle）**
- **判断/分支**：**菱形（diamond）**
- **流向**：箭头（arrow），只有一条流出主线
- **常见错误**：判断节点后缺"No"分支、菱形文字溢出

## 4. 时序图（Sequence Diagram）[UML]
- **生命线**：**虚线（dashed）** 从参与者向下延伸
- **参与者**：矩形框在顶部
- **消息**：实线箭头（同步），虚线箭头（返回）
- **激活框**（可选）：细长矩形覆盖在生命线上表示处理期间
- **常见错误**：生命线用实线（这是 UML 1.x 的旧习惯，UML 2 允许但推荐虚线）

## 5. ER 图（Entity-Relationship）[Crow's Foot]
- **实体**：矩形，顶部实体名，下方属性列表（PK 标记 `*` 或下划线）
- **关系线**：实体之间的连线，标注动词短语
- **基数标记**（Crow's Foot 核心）：
  - `‖` = 恰好一个（mandatory one）
  - `○‖` = 零或一个（optional one）
  - `‖<` = 一到多个（mandatory many，< 是三叉"crow's foot"）
  - `○<` = 零到多个（optional many）
- **M:N 关系**：必须用关联实体拆成两个 1:M
- **常见错误**：缺基数标记（只有线没有标记是最常见的业余画法）

## 6. 状态机（State Machine）[UML]
- **状态**：**圆角矩形（rectangle + roundness:{type:3}）**，not 椭圆！
- **初始状态**：实心圆
- **终止状态**：实心圆外套圆环
- **转换**：实线 + 开口箭头，标注 `事件[条件]/动作`
- **常见错误**：用椭圆表示状态（椭圆是流程图开始/结束，不是状态机状态）

## 7. 思维导图
- **中心**：一个节点（通常是椭圆或圆角矩形）
- **分支**：从中心向外放射，用线连接
- **层级**：一级分支用粗线，二级更细
- **常见错误**：分支太多（>7 个一级分支）、深度太浅

## 8. 网络拓扑
- **区域**：虚线分组（DMZ / 内网 / 外网）
- **设备**：矩形框，标注设备名和 IP
- **防火墙**：通常放在边界位置
- **常见错误**：缺少安全层级（直接内网连外网）

## 9. 用户旅程图（User Journey Map）
- **纵向阶段**：5-7 个阶段（认知→考虑→决策→首次使用→日常使用→留存→推荐）
- **行**：每个阶段下列出：触点、动作、情绪（😊/😐/😣）、痛点、机会
- **情绪曲线**：折线连接各阶段情绪高低点
- **常见错误**：只有阶段和触点，缺痛点/机会（没有 actionable insight 的旅程图是废的）

## 10. 组件图（Component Diagram）
- **组件**：矩形，标注组件名和职责
- **接口**：⊙ 或 lollipop 表示 provided，半圆弧表示 required
- **依赖**：虚线箭头
- **常见错误**：组件粒度过细（把 class 当组件画）
